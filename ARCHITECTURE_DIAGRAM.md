## Key Design Decisions

### 1. **Two-Tier Retrieval**
- **Rationale**: Balance between precision and recall
- **Primary tier** (0.5): High-quality results
- **Fallback tier** (0.4): Broader coverage when needed

### 2. **Confidence Indicators**
- **Rationale**: Transparency and user trust
- **Implementation**: Color-coded visual feedback
- **Benefit**: Users can assess response reliability

### 3. **Async Operations**
- **Rationale**: Responsive UI during long operations
- **Implementation**: asyncio with run_in_executor
- **Benefit**: Immediate feedback, non-blocking

### 4. **Intent Classification**
- **Rationale**: Handle ambiguous queries proactively
- **Implementation**: Separate LLM call before retrieval
- **Benefit**: Better user experience, clearer queries

### 5. **Low Temperature (0.1)**
- **Rationale**: Minimize hallucination, maximize factuality
- **Trade-off**: Less creative but more grounded
- **Result**: More deterministic, context-faithful responses

### 6. **Enhanced Prompts**
- **Rationale**: Guide LLM to stay grounded in context
- **Implementation**: Explicit anti-hallucination rules
- **Limitation**: Cannot fully prevent hallucination (89.7% rate)

## System Limitations

1. **High Hallucination Rate (89.7%)**
   - LLM generates information beyond context
   - Cannot be fully solved with prompt engineering
   - Requires fine-tuning or constrained generation

2. **Low Faithfulness (10.3%)**
   - Responses deviate from retrieved context
   - Related to hallucination problem
   - Documented research finding

3. **Intent Classification Sensitivity**
   - May mark clear queries as AMBIGUOUS
   - Trade-off between clarity and UX

4. **Retrieval Quality Dependency**
   - System quality depends on vector search results
   - Limited by dataset coverage
   - Similarity thresholds are heuristic
